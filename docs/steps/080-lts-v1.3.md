# Step 080 — LTS v1.3

## Phase
Phase 8 — LTS Release

## Purpose
Tag v1.3 LTS. This release restores the solver pipeline to a working state after a chain of 5 bugs that had been silently breaking it. The solver now produces real draft schedules with assignments. This is a foundation release — the schedule quality is functional but not yet optimal.

## What changed since v1.2

### Solver pipeline — fully restored
The entire solver pipeline was broken. Triggering a run produced either an empty draft or a run stuck in `Running` forever. Five bugs were found and fixed:

1. **Hard headcount constraint** (Python) — `>=` constraint made model INFEASIBLE when any slot couldn't be fully staffed. Changed to soft coverage objective only.
2. **Blocked presence states ignored** (Python) — `blocked` and `on_mission` states were not enforced. Only `at_home` was checked.
3. **Normalizer crashed in tests** (.NET) — `ExecuteSqlRawAsync` called unconditionally, failing on in-memory DB.
4. **Redis deserialization failure** (.NET) — `SolverJobMessage` deserialization failed due to missing `PropertyNameCaseInsensitive = true`, crashing the worker silently on every job.
5. **`stability_score` column overflow** (PostgreSQL) — `NUMERIC(5,2)` overflowed with real solver output (values 2000–8000+). Widened to `NUMERIC(18,2)`.

### New tests
- 8 solver end-to-end tests hitting the live Python solver directly
- 5 pipeline integration tests: normalizer → solver → result verification

### Migration
- `026_fix_stability_score_precision.sql` — widens `stability_score` column

## Known limitations (not a good version yet)
- Schedule quality is basic — fairness, rest constraints, and role/qualification filtering need tuning
- Personal and role constraints spec not yet implemented
- Partial schedules (uncovered slots) are not surfaced clearly in the UI
- No admin review flow for partial results
